fix: keep ModMixer's own instrumentation + engine specs OUT of the reorder fold

THE 2026-06-13 EVERY-OTHER-BOOT BRICK (sequel to f97adce). After f97adce the
reorder chain worked on a clean boot, then bricked on the next, alternating.
Diagnosis from the recovery-boot log:
- log line 682: AUTO SAFE MODE fired -> the watchdog caught that the prior boot
  applied levers but never confirmed a healthy world (the brick) and disabled
  levers this boot. So the "clear" boots are safe-mode boots with reorders OFF;
  the underlying brick still fires on lever boots. Watchdog working as designed.
- census line 8331 (even in safe mode): Drivable.updateVehiclePhysics is hooked
  by EnhancedVehicle, AdvancedDamageSystem, soundExpansionMP AND FS25_0_ModMixer.
  Our OWN instrumentation wrappers (hook-probe timers + exec markers) ride these
  physics chains, and batch-4 fenv-fingerprinting now names them as ours.

ROOT CAUSE: the recent self-hide (ec9f4ee/2c8df2f) hid ModMixer's own hooks from
the contest VIEWS but not from the reorder BUFFER. So the fold on a reordered
physics target now includes our referee wrapper as if it were a contestant. Folding
our wrapper -- or an engine per-type spec overwrite like AIAutomaticSteering, which
routes through Utils.overwrittenFunction when it wraps the inherited class fn -- into
the base-class chain mis-wires `self`; the per-type AIAutomaticSteering overwrite then
reads self.spec_aiAutomaticSteering == nil every frame -> no controls (the same
symptom as f97adce, different source). Intermittent because whether our wrapper / a
spec fn lands in a given boot's buffer depends on per-boot attribution + install order.

FIX: the reorder fold takes CONTESTANTS ONLY. A hook is foldable iff it belongs to a
real third-party mod (mod ~= nil and mod ~= _selfMod). Excluded:
  - mod == _selfMod  -> ModMixer's own instrumentation. The referee never plays; the
    UI already hides these (SHOW_SELF_HOOKS=false), now the fold excludes them too.
  - mod == nil       -> engine/spec composition or unattributed. Per-type spec
    overwrites must stay in their per-type slot, never folded into the class chain.
Excluded hooks fall through and install NORMALLY (still run, just never reordered),
logged once per target ("REORDER: NOT folding ... referee, not a contestant"). _selfMod
is captured at load (g_currentModName == us while this file executes first). buf.base
stays pristine; the clean fold of named contestants is planted via classMethodSet, so
AIAutomaticSteering's super resolves to it with correct self -> no nil, no brick.
Reorder still works for genuine mod-vs-mod chains (boot 1 proved the clean case).

Verified: lupa battery ALL OK, TestRunner PASS. Watchdog + MODMIXER_SAFE_MODE.txt
remain the recovery net if any other lever ever bricks.
